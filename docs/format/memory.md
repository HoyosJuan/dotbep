# memories/

The `memories/` folder inside a `.bep` archive stores **project memories** — structured notes generated by an LLM from workflow instance data. They capture durable patterns and institutional learnings that are not encoded in the structured data of `bep.json`.

## Structure

```
memories/
├── index.json          ← metadata array for all memories in this BEP
├── {slug}.md           ← markdown body for each memory
└── {slug}.md
```

`index.json` holds the metadata. Each entry's `id` doubles as the filename of its body file (`memories/{id}.md`).

## Fields

| Field | Type | Description |
|---|---|---|
| `id` | `string` | Kebab-case slug. Serves as the filename (`memories/{id}.md`). Stable — does not change when `name` changes. |
| `name` | `string` | Short display title (max 6 words). Can be updated freely. |
| `type` | `"realization" \| "pattern"` | Category of memory (see below). |
| `links` | `string[]` | Slugs of other memories in this BEP that this memory relates to. |
| `createdAt` | ISO datetime | When the memory was first written. |
| `updatedAt` | ISO datetime | When the memory was last updated. |
| `data` | `Record<string, unknown>` (optional) | Free-form metadata authored by the LLM (e.g. `confidence`, `evidenceCount`). No enforced structure. |

## Memory types

### `realization`

An institutional learning of the project — something structurally significant the LLM detected that should change future behavior. Realizations speak in the voice of the project, not of any individual actor.

### `pattern`

A recurring behavior or correlation observed across workflow instances. Includes behavioral patterns tied to specific actors, roles, or nodes.

## Versioning

`memories/` is **not versioned**. Neither `index.json` nor the individual `.md` files are included in version diffs. The BEP history does not track changes to this folder.

## Relationship to `skills/`

| | `memories/` | `skills/` |
|-|-------------|-----------|
| Content | What has happened / what was learned | How the LLM should act |
| Set by | LLM (from workflow instance data) | BIM Manager (at project start) |
| Changes | Grows and evolves with the project | Stable and prescriptive |

## IDs as slugs

Memory IDs are slugs rather than UUIDs so that cross-references between memories are readable in plain markdown without any resolution step. An LLM authoring a memory can reference a related one by slug directly in the body, and a human reading the raw file understands the link immediately.

See `core/adrs/001-memory-slug-ids.md` for the full decision record.
