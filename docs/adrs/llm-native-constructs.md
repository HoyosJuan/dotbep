# ADR — LLM-native constructs in the `.bep` format: skills, memories, and flags

## Context

The `.bep` format primarily models structured project-management data — teams, deliverables, workflows — authored through a validated schema. Some content produced while working with a BEP doesn't fit that shape naturally: free-form guidance text, narrative knowledge distilled from execution, and diagnostic signals meant to drive an agent's next action.

## Decision

**The format includes constructs designed to be authored and consumed directly by an LLM, using the representation natural to that purpose rather than forcing it into the same shape as human-authored plan data.**

- **Skills** (`skills/{name}/SKILL.md` + `skills/{name}/resources/`) — free-form markdown, unvalidated by the schema.
- **Memories** (`memories/{id}.md`) — narrative markdown generated by an LLM from workflow execution.
- **`Flag`** — a schema entity, but one generated automatically by analysis rather than authored by a user through the plan's normal edit flow.

## Reasoning

Prose that an LLM generates (a skill's instructions, a memory's narrative) has no natural fit inside a validated JSON schema — forcing it into structured fields would fight the medium for no benefit, since nothing downstream needs to query into its structure. Plain markdown is exactly the shape an LLM produces and reads best.

`Flag` is different: it is structured, because a diagnostic signal (severity + message, tied to a specific entity) is genuinely queryable data, not prose. But it is still LLM-native in origin — generated by analysis of the BEP's state, not by a user filling out a form — which is precisely why it must not be treated as user-authored plan data (see Consequences).

## Alternatives discarded

**Model skill/memory content as structured schema fields** — rejected because it would force generated prose into a rigid shape with no consumer that needs to query into it, adding validation overhead with no payoff.

**Treat `flags` as ordinary user-authored plan data** — rejected because flags are regenerated by analysis, not intentional edits; including them in the plan's diff/versioning flow would surface noise unrelated to real authored changes.

## Consequences

- `skills/` and `memories/` files are plain text, unvalidated by the Zod schema — any structure within their content is the LLM's responsibility, not core's.
- `flags` is excluded from `arrayDefs` and the diff/versioning flow (`src/utils/diff.ts`) — it never appears as a pending plan change.
- Any future LLM-native addition to the format should ask the same question this ADR resolves: does this content have a natural human-authored shape, or is it something an LLM produces/consumes directly — and if the latter, it should get this same free-form-or-diagnostic treatment, not be forced into the plan schema's mold.
