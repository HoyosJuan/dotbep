# ADR 001 — Memory IDs as slugs

## Context

The `memories/` system stores project memories generated by an LLM from workflow instance data. Each memory consists of a metadata entry in `memories/index.json` and a markdown body in `memories/{id}.md`. Memories can reference other memories in the project.

Two ID formats were considered: UUIDs and human-readable slugs.

## Decision

Memory IDs are **slugs** — short, kebab-case strings that describe the pattern the memory captures (e.g. `l-chen-cuello-de-botella-mep`, `cancelaciones-sin-nota-explicativa`).

## Reasoning

An LLM writing a memory may want to reference related memories directly within the markdown body (e.g. "this pattern is a consequence of what is described in `[[acceso-acc-permisos-fuente-de-retraso]]`"). With slugs, the reference is readable without any resolution — a human reading the raw markdown understands the link immediately. With UUIDs, the reference would be opaque regardless of whether a rendering tool replaces it with the memory name.

The `id` field also serves as the filename (`memories/{id}.md`), so legibility extends to the file system as well.

## Alternatives discarded

**UUIDs** — would make rename operations safe (changing a memory's name wouldn't break references), but at the cost of opaque identifiers in markdown content and filenames. Since memories are rarely renamed and references are authored by an LLM that reads the index, the rename safety benefit does not outweigh the readability cost.

## Consequences

- The LLM is responsible for generating a stable, descriptive slug when creating a memory.
- If a memory is renamed, its ID (slug) should remain unchanged to preserve existing references. The `name` field in `index.json` holds the display title and can change freely.
- Slug uniqueness within a BEP is enforced by the `add()` operation.
